## 1. 서비스 개요

독거 어르신의 스마트워치에서 수집된 건강 데이터를 가족 구성원 전원이 동시에 확인하고, 이상 감지 알림이 가족 채팅 및 역할 분담으로 자연스럽게 연결되는 가족 공동 케어 앱이다. 어르신용 앱과 가족(보호자)용 앱을 분리하여 제공하며, 어르신은 건강 수치 대신 '오늘 가족 3명이 확인했어요'와 같은 연결감 중심의 화면을 본다. 삼성헬스 API를 통해 활동량·심박수·수면 데이터를 수신하고, Android(Kotlin) 기반으로 개발한다.

---

## 2. 문제 정의

현재 독거 어르신 돌봄은 아래 세 가지 구조적 문제를 갖는다.

1. **돌봄 집중**: 자녀 중 한 명(주로 장남·장녀)이 모든 알림·연락을 떠안아 번아웃 발생
2. **정보 단절**: 가족 구성원 간 어르신 건강 정보가 공유되지 않아 중복 연락 또는 완전 공백 발생
3. **이상 감지 지연**: 어르신이 쓰러지거나 이상이 생겨도 몇 시간 뒤에야 발견되는 상황

기존 대안(1:1 스마트워치 알림 앱, 복지사 주 1회 방문)은 가족 간 공동 대응 구조를 제공하지 못한다.

## 3. 타깃 사용자

| 구분 | 대상 | 핵심 니즈 |
|------|------|-----------|
| 1차 타깃 | 독거 어르신을 둔 가족 보호자 (30~50대 자녀) | 돌봄 부담 분산, 실시간 이상 알림, 가족 간 조율 |
| 2차 타깃 | 스마트워치 착용 독거 어르신 (60~80대) | 자녀에게 짐이 되지 않으면서 연결감을 느끼고 싶음 |

- **페르소나 A** (42세 직장인): 지방 혼자 사시는 어머니가 걱정되지만 매일 전화는 부담. 형제 간 '네가 해' 눈치 싸움이 반복됨.
- **페르소나 B** (74세 어르신): 아파도 자녀 짐이 될까 '괜찮다'고 함. 혼자인 밤이 두렵지만 연락하기 곤란하다고 느낌.

## 4. 핵심 가치 제안

> **혼자 짊어지던 부모님 걱정을, 가족 모두가 함께 나누는 케어 앱**

- **돌봄 분산**: 한 명이 지쳐 포기하는 돌봄 → 가족이 나눠 오래 지속하는 돌봄
- **데이터의 재해석**: 워치 건강 수치가 숫자가 아닌 '가족이 함께 행동하는 계기'가 됨
- **어르신 자율성 보장**: 어르신이 공개 범위를 직접 설정 → '감시'가 아닌 '내가 선택한 연결'

---

## 5. 사용자 시나리오

**시나리오 1 — 이상 감지 → 가족 공동 대응**

오후 2시, 어머니 워치에서 '2시간 이상 움직임 없음' 감지 → 가족 앱 전체에 알림 발송 → 채팅방에서 딸이 '내가 전화해볼게' → 전화 후 '주무시고 계셨어요' 공유 → 상황 자연스럽게 종료.

**시나리오 2 — 주간 역할 자동 제안**

월요일 아침, 앱이 이번 주 케어 역할을 일정에 맞게 자동 제안 → 아들 월·수 전화 / 딸 화·목 전화 / 며느리 금요일 방문 → 각자 수락. 형제 간 역할 갈등 사라짐.

**시나리오 3 — 어르신의 연결 확인**

어르신이 어르신용 앱을 열면 '오늘 아들·딸·손자 3명이 확인했어요' 표시 → 딸 메시지 '내일 방문할게요' 확인 → 안도감을 느낌.

## 6. 핵심 기능 요약

| 기능 | 대상 | 설명 | MVP 포함 |
|------|------|------|----------|
| 건강 데이터 공유 대시보드 | 가족(보호자) | 삼성헬스 데이터를 가족 전원이 동일하게 조회 | ✅ |
| 이상 감지 알림 | 가족(보호자) | 활동 없음·심박 이상 등 가족 전체에 Push 알림 | ✅ |
| 알림 연동 채팅방 | 가족(보호자) | 알림 수신 후 해당 알림과 연결된 채팅 스레드 | ✅ |
| 어르신 확인 화면 | 어르신 | '오늘 N명이 확인했어요' + 가족 메시지 수신 | ✅ |
| 주간 역할 자동 제안 | 가족(보호자) | 케어 역할을 일정 기반으로 자동 배분·수락 | 추후 |
| 약 복용 여부 알림 | 전체 | 알약 복용 체크 및 알림 (제약 조건 복잡) | 제외 |

## 7. 경쟁 서비스 대비 차별점

| 비교 항목 | 기존 1:1 알림 앱 | 본 서비스 |
|-----------|-----------------|-----------|
| 알림 수신자 | 담당자 1명 | 가족 전원 동시 수신 |
| 알림 → 채팅 연결 | 없음 | 알림이 채팅 트리거로 자동 연결 |
| 역할 분담 기능 | 없음 | 케어 역할 자동 제안 및 수락 구조 |
| 어르신 화면 | 건강 수치 그대로 표시 | '가족이 확인했어요' 연결감 중심 UI |
| 어르신 자율성 | 없음 | 공개 범위 직접 설정 가능 |

## 8. 비즈니스 모델 개요

수익 모델은 B2G(정부·공공기관 대상 판매)를 기반으로 한다. 지자체 복지과, 보건복지부, 건강보험공단, 사회서비스원 등이 주요 비용 지불 주체다. 공공기관의 도입 유인은 고독사 예방 비용 절감 및 복지사 업무 효율화다. 향후 건강보험공단 연계를 통해 건강 데이터 기반 예방 의료 서비스로 확장 가능하다.

---

## 9. MVP 범위

이번 학기 핵심 목표: 가족 공동 케어 앱의 핵심 플로우 시연 가능한 수준의 동작 앱 구현

| 구분 | 항목 | 비고 |
|------|------|------|
| 포함 | 건강 데이터 공유 대시보드 (가족 화면) | 삼성헬스 연동 또는 더미 데이터 |
| 포함 | 이상 감지 알림 (활동 없음, 심박 이상) | In-App Push 알림 |
| 포함 | 알림 연동 채팅방 (가족 공동 대화) | 기본 채팅 UI |
| 포함 | 어르신용 '가족이 확인했어요' 화면 | 어르신 전용 앱 화면 |
| 제외 | 약 복용 여부 알림 | 제약 조건 복잡, 전문 지식 필요 |
| 제외 | 로그인 기능 | MVP 시연 범위 초과 |
| 제외 | 주간 역할 자동 제안 | 추후 기능으로 이동 |

발표 시연 장면: (1) 가족들이 같은 건강 데이터를 보는 화면, (2) 이상 알림 수신 후 채팅 연결, (3) 어르신이 '가족이 확인했어요'를 보는 화면
