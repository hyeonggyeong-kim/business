# requirements.md

> **프로젝트명**: 가족 공동 케어 앱 (Family Care)
> **팀명**: 7팀 | **팀장**: 정승혁 | **팀원**: 구민우, 김형경
> **기준**: MVP (이번 학기 시연 범위 기준)
> **마지막 수정일**: 2026-04-10

---

## 1. 문서 목적

이 문서는 가족 공동 케어 앱이 MVP 단계에서 반드시 충족해야 하는 기능적·비기능적 요구사항을 정의한다. 개발 범위를 명확히 하고, 팀원 간 기준을 통일하며, 발표 시연에 필요한 최소 요건을 확정하는 것이 목적이다.

**중요도 기준**

| 표기 | 의미 |
|------|------|
| **Must** | MVP에 반드시 포함. 없으면 시연 불가 |
| **Should** | 있으면 완성도가 높아짐. 가능하면 구현 |
| **Could** | 향후 확장 시 고려. 이번 학기 미포함 |

## 2. 사용자 유형별 요구사항

### 2-1. 보호자 (Guardian) — 어르신의 자녀 또는 가족 구성원

| ID | 요구사항 | 중요도 | 사용자 스토리 |
|----|---------|--------|--------------|
| G-REQ-01 | 보호자는 어르신의 최신 건강 데이터(심박수, 활동량, 수면 시간)를 하나의 화면에서 확인할 수 있어야 한다 | **Must** | 지방에 사는 딸로서, 어머니의 상태를 앱 하나로 빠르게 확인하고 싶다 |
| G-REQ-02 | 보호자는 어르신에게 이상이 감지됐을 때 푸시 알림을 받을 수 있어야 한다 | **Must** | 직장에 있을 때도 어머니 이상 상황을 즉시 알고 싶다 |
| G-REQ-03 | 보호자는 알림 수신 후 가족 채팅방으로 바로 이동할 수 있어야 한다 | **Must** | 알림을 받은 뒤 가족과 빠르게 상황을 공유하고 싶다 |
| G-REQ-04 | 보호자는 가족 채팅방에서 메시지를 주고받을 수 있어야 한다 | **Must** | 누가 전화할지 채팅으로 자연스럽게 조율하고 싶다 |
| G-REQ-05 | 보호자는 이번 주 케어 역할 분담 제안을 확인하고 수락하거나 수정할 수 있어야 한다 | **Should** | 매번 형제들과 '네가 해라' 눈치 싸움 없이 역할을 나누고 싶다 |
| G-REQ-06 | 보호자는 과거 이상 알림 이력을 날짜 순으로 확인할 수 있어야 한다 | **Should** | 어머니 건강 패턴이 최근에 어떻게 변했는지 돌아보고 싶다 |
| G-REQ-07 | 보호자는 알림의 대응 완료 여부를 표시할 수 있어야 한다 | **Should** | 이미 해결된 알림과 아직 확인 중인 알림을 구분하고 싶다 |

### 2-2. 어르신 (Elder) — 독거 어르신, 스마트워치 착용자

| ID | 요구사항 | 중요도 | 사용자 스토리 |
|----|---------|--------|--------------|
| E-REQ-01 | 어르신은 오늘 가족 몇 명이 자신의 상태를 확인했는지 메인 화면에서 볼 수 있어야 한다 | **Must** | 자녀들이 나를 생각해주고 있다는 것을 느끼고 싶다 |
| E-REQ-02 | 어르신은 가족이 보낸 메시지를 확인할 수 있어야 한다 | **Must** | 딸이 내일 방문한다고 했는지 앱에서 확인하고 싶다 |
| E-REQ-03 | 어르신은 가족에게 간단한 답장을 보낼 수 있어야 한다 | **Should** | 아들 메시지에 짧게 '알겠어'라고 답하고 싶다 |
| E-REQ-04 | 어르신은 어떤 건강 데이터를 가족과 공유할지 직접 선택할 수 있어야 한다 | **Should** | 내 건강 정보를 내가 통제하고 싶다. 감시받는 느낌은 싫다 |
| E-REQ-05 | 어르신 앱의 모든 글씨는 충분히 크게 표시되어야 한다 (최소 18sp 이상) | **Must** | 노안이 있어서 작은 글씨는 읽기 어렵다 |

### 2-3. Wear OS (어르신 스마트워치)

| ID | 요구사항 | 중요도 | 사용자 스토리 |
|----|---------|--------|--------------|
| W-REQ-01 | 워치 앱은 Samsung Health API를 통해 심박수, 활동량 데이터를 수집해 서버로 전송할 수 있어야 한다 | **Must** | 워치를 차고 있으면 자동으로 내 건강 데이터가 가족에게 전달되길 원한다 |
| W-REQ-02 | 워치는 가족이 건강 데이터를 확인했을 때 간단한 알림을 표시할 수 있어야 한다 | **Should** | 워치에서 아들이 확인했다는 메시지를 보고 싶다 |

## 3. 핵심 기능 요구사항

### 3-1. 건강 데이터 수집 및 공유

| ID | 요구사항 | 중요도 |
|----|---------|--------|
| F-REQ-01 | 시스템은 Samsung Health API로부터 심박수, 걸음 수, 수면 시간 데이터를 수신할 수 있어야 한다 | **Must** |
| F-REQ-02 | 수집된 건강 데이터는 해당 어르신의 가족 그룹 전원에게 동일하게 표시되어야 한다 | **Must** |
| F-REQ-03 | 건강 데이터는 가족 대시보드 화면에서 가장 최근 데이터 기준으로 표시되어야 한다 | **Must** |
| F-REQ-04 | Samsung Health API 연동이 불가한 경우, 더미 데이터로 화면이 동작해야 한다 (시연 대비) | **Must** |

### 3-2. 이상 감지 및 알림

| ID | 요구사항 | 중요도 |
|----|---------|--------|
| F-REQ-05 | 시스템은 어르신의 활동이 2시간 이상 감지되지 않을 경우 이상으로 판단하고 알림을 발송해야 한다 | **Must** |
| F-REQ-06 | 시스템은 심박수가 정상 범위(분당 40~120회)를 벗어날 경우 이상으로 판단하고 알림을 발송해야 한다 | **Must** |
| F-REQ-07 | 이상 감지 알림은 가족 그룹 내 보호자 전원에게 동시에 발송되어야 한다 | **Must** |
| F-REQ-08 | 알림에는 이상 유형(활동 없음 / 심박 이상 / 수면 이상)이 명시되어야 한다 | **Must** |

### 3-3. 가족 채팅

| ID | 요구사항 | 중요도 |
|----|---------|--------|
| F-REQ-09 | 보호자는 가족 채팅방에서 텍스트 메시지를 전송하고 수신할 수 있어야 한다 | **Must** |
| F-REQ-10 | 이상 알림 상세 화면에는 '채팅방으로 이동' 버튼이 있어야 한다 | **Must** |
| F-REQ-11 | 알림에서 연결된 채팅 메시지는 해당 알림과 연결됐음을 표시해야 한다 | **Should** |
| F-REQ-12 | 어르신도 가족이 보낸 메시지를 읽고 간단한 답장을 보낼 수 있어야 한다 | **Should** |

### 3-4. 역할 분담

| ID | 요구사항 | 중요도 |
|----|---------|--------|
| F-REQ-13 | 시스템은 매주 가족 구성원의 케어 역할(전화/방문 담당일)을 자동으로 제안해야 한다 | **Should** |
| F-REQ-14 | 보호자는 자신에게 배정된 역할을 수락하거나 변경을 요청할 수 있어야 한다 | **Should** |
| F-REQ-15 | 전원 수락 시 역할 분담이 확정되어야 하고, 미수락 시 리마인드 알림을 발송해야 한다 | **Could** |

### 3-5. 어르신 확인 현황

| ID | 요구사항 | 중요도 |
|----|---------|--------|
| F-REQ-16 | 어르신 앱 메인 화면에는 오늘 가족 몇 명이 건강 데이터를 확인했는지 표시되어야 한다 | **Must** |
| F-REQ-17 | 어르신 앱은 건강 수치 대신 "오늘 N명이 확인했어요"와 같은 감성적 메시지로 표현해야 한다 | **Must** |
| F-REQ-18 | 어르신 앱에는 가족의 방문 예정일 또는 메시지가 표시되어야 한다 | **Should** |

## 4. 예외 상황 및 오류 처리 요구사항

| ID | 상황 | 요구사항 | 중요도 |
|----|------|---------|--------|
| E-REQ-01 | Samsung Health API 연동 실패 | 더미 데이터를 자동으로 로드하여 화면이 빈 상태로 표시되지 않아야 한다 | **Must** |
| E-REQ-02 | 네트워크 연결 없음 | 마지막으로 받은 데이터를 로컬 캐시에서 표시하고, "인터넷 연결을 확인해주세요" 문구를 보여줘야 한다 | **Should** |
| E-REQ-03 | 푸시 알림 미수신 | 알림 목록 화면에서 수동 새로고침으로 최신 알림을 조회할 수 있어야 한다 | **Should** |
| E-REQ-04 | 채팅 메시지 전송 실패 | "메시지 전송 실패" 토스트 메시지를 노출하고, 재전송 버튼을 제공해야 한다 | **Should** |
| E-REQ-05 | 워치 앱과 폰 앱 간 연결 끊김 | 보호자 앱에서 "워치 연결 끊김" 상태를 대시보드에 표시해야 한다 | **Could** |
| E-REQ-06 | 이상 알림 중복 발송 | 동일 조건으로 1시간 이내 중복 알림이 발송되지 않아야 한다 | **Should** |

## 5. 비기능 요구사항

### 5-1. 성능

| ID | 요구사항 | 중요도 |
|----|---------|--------|
| NF-PERF-01 | 보호자 앱의 건강 대시보드는 앱 실행 후 3초 이내에 데이터를 표시해야 한다 | **Must** |
| NF-PERF-02 | 이상 감지 알림은 이상 발생 후 30초 이내에 보호자 전원에게 도달해야 한다 | **Must** |
| NF-PERF-03 | 채팅 메시지는 전송 후 3초 이내에 상대방 화면에 표시되어야 한다 | **Should** |
| NF-PERF-04 | 워치에서 건강 데이터 전송 주기는 최대 5분 간격을 초과하지 않아야 한다 | **Should** |

### 5-2. 보안

| ID | 요구사항 | 중요도 |
|----|---------|--------|
| NF-SEC-01 | 건강 데이터는 인가된 가족 그룹 구성원만 조회할 수 있어야 한다 | **Must** |
| NF-SEC-02 | 서버와 클라이언트 간 통신은 HTTPS를 사용해야 한다 | **Must** |
| NF-SEC-03 | 어르신의 개인정보(이름, 건강 데이터)는 외부에 노출되지 않아야 한다 | **Must** |
| NF-SEC-04 | MVP 단계에서 로그인 기능은 제외하나, 그룹 ID 기반의 최소한의 접근 분리는 유지해야 한다 | **Must** |

### 5-3. 사용성

| ID | 요구사항 | 중요도 |
|----|---------|--------|
| NF-UX-01 | 어르신 앱은 화면당 선택 가능한 버튼이 3개를 초과하지 않아야 한다 | **Must** |
| NF-UX-02 | 보호자 앱에서 알림 수신 후 채팅방 진입까지 최대 2번의 탭으로 이동할 수 있어야 한다 | **Must** |
| NF-UX-03 | 모든 핵심 기능은 처음 사용하는 사람도 별도 설명 없이 10분 내 사용할 수 있을 수준이어야 한다 | **Should** |
| NF-UX-04 | 로딩 중에는 스켈레톤 UI 또는 프로그레스 인디케이터를 표시해야 한다 | **Should** |

### 5-4. 접근성

| ID | 요구사항 | 중요도 |
|----|---------|--------|
| NF-ACC-01 | 어르신 앱의 기본 텍스트 크기는 18sp 이상이어야 한다 | **Must** |
| NF-ACC-02 | 어르신 앱의 터치 타겟(버튼, 링크)은 최소 48dp × 48dp 이상이어야 한다 | **Must** |
| NF-ACC-03 | 중요 정보는 색상만으로 구분하지 않고 텍스트 또는 아이콘을 함께 사용해야 한다 | **Should** |

### 5-5. 유지보수성

| ID | 요구사항 | 중요도 |
|----|---------|--------|
| NF-MAINT-01 | 이상 감지 임계값(활동 없음 시간, 심박 범위)은 코드 내 상수로 명시적으로 관리해야 한다 | **Must** |
| NF-MAINT-02 | 더미 데이터와 실제 API 데이터는 플래그 하나로 전환 가능한 구조여야 한다 | **Must** |
| NF-MAINT-03 | 각 모듈(건강 데이터, 알림, 채팅, 역할 분담)은 독립적으로 개발·테스트 가능한 구조여야 한다 | **Should** |

## 6. MVP 포함 범위

> 이번 학기 발표 시연에서 반드시 동작해야 하는 기능

| 번호 | 기능 | 관련 요구사항 ID |
|------|------|----------------|
| 1 | 어르신 건강 데이터 수집 (더미 데이터 포함) | F-REQ-01, F-REQ-04 |
| 2 | 가족 전체 건강 데이터 공유 대시보드 (보호자 앱) | G-REQ-01, F-REQ-02, F-REQ-03 |
| 3 | 이상 감지 알림 발송 (활동 없음 / 심박 이상) | F-REQ-05, F-REQ-06, F-REQ-07, F-REQ-08 |
| 4 | 알림 → 가족 채팅방 연결 | G-REQ-02, G-REQ-03, F-REQ-09, F-REQ-10 |
| 5 | 어르신 앱 메인 화면 (가족 확인 현황 표시) | E-REQ-01, F-REQ-16, F-REQ-17 |
| 6 | 어르신 앱 가족 메시지 확인 화면 | E-REQ-02 |

## 7. MVP 제외 범위

> 이번 학기에는 구현하지 않음. 향후 버전에서 추가 예정

| 제외 기능 | 제외 이유 |
|-----------|----------|
| 로그인 / 회원가입 | 인증 로직 복잡도 과다, 시연과 무관 |
| 약 복용 알림 | 약학적 전문 지식 및 안전 검증 필요 |
| 역할 분담 자동 확정 및 리마인드 | 핵심 플로우 외 부가 기능 |
| 관리자(지자체) 웹 대시보드 | B2G는 향후 단계, 이번 학기 범위 초과 |
| 건강보험공단 API 연동 | 공공 API 승인 절차 불가 |
| Wear OS 알림 표시 | 워치 UI 구현보다 데이터 수집에 집중 |
| 채팅 메시지 재전송 UI | 에러 처리 고도화는 추후 |
| 다수 어르신 관리 구조 | 현재 1가족 1어르신 구조로 충분 |

## 8. 성공 기준

> MVP 발표 시연에서 아래 기준을 모두 만족해야 한다

| 번호 | 성공 기준 | 확인 방법 |
|------|----------|----------|
| SC-01 | 시나리오 1 완전 재현 가능: 워치 이상 감지 → 보호자 앱 알림 수신 → 채팅방 이동 → 메시지 전송 | 시연 라이브 데모 |
| SC-02 | 시나리오 3 완전 재현 가능: 보호자가 어르신 데이터 확인 → 어르신 앱에 확인자 수 반영 | 시연 라이브 데모 |
| SC-03 | 보호자 앱과 어르신 앱이 동일한 건강 데이터를 표시 | 두 기기 화면 동시 비교 |
| SC-04 | 더미 데이터 사용 시에도 모든 화면이 빈 상태 없이 정상 표시됨 | UI 스크린샷 확인 |
| SC-05 | 어르신 앱에서 글씨 크기 및 버튼 크기가 접근성 기준을 충족함 | 직접 조작 확인 |
| SC-06 | 팀원 외 타인이 앱을 처음 보고 핵심 흐름을 별도 설명 없이 이해 가능 | 발표 현장 질의응답 |
